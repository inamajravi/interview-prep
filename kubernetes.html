Core Objects — Pod, Deployment, Service

The three building blocks nearly everything else sits on top of
Pod

Define: The smallest deployable unit in Kubernetes — one or more tightly coupled containers that always run together on the same node, sharing network and storage. Analogy: A pod is like a shared apartment — the containers inside it share the same address (IP) and utilities (network/volumes), even if they're separate "tenants" (e.g. a main app container plus a logging sidecar).

Why usually more than one container: A "sidecar" pattern — a helper container (log shipper, proxy, cert renewer) that needs to live and die alongside the main container, sharing its network namespace.

Deployment

Define: A controller that manages a set of identical Pods — declares "I want N replicas of this Pod spec running," and continuously reconciles reality toward that desired state, handling rolling updates and rollbacks. Analogy: A Deployment is a restaurant manager who always makes sure exactly 5 tables are staffed — if a server calls in sick (a Pod crashes), the manager immediately calls in a replacement, without you having to notice or intervene.

Service

Define: A stable network identity (a fixed DNS name/IP) in front of a changing set of Pods. Pods are ephemeral — they get recreated with new IPs constantly — so nothing should ever talk to a Pod IP directly. A Service is the thing that stays constant while Pods behind it come and go.

Analogy: A Service is a company's general support phone number — it never changes, even though the actual support agent who picks up (the Pod) rotates constantly. Callers never need the agent's personal number.
Trade-off / why not just run containers directly
What you gainSelf-healing, declarative desired-state management, and stable networking despite constantly churning Pod IPs.
What it costsReal operational complexity — networking, RBAC, and resource management all become Kubernetes-specific concerns your team needs to own.
Say this in interview

"The core insight underneath Pods, Deployments, and Services is that Kubernetes assumes Pods are disposable and will die — the Deployment's job is replacing them without anyone noticing, and the Service's job is giving callers something stable to talk to despite that constant churn underneath."

Scaling & Self-Healing

Liveness/Readiness probes, and the Horizontal Pod Autoscaler
Liveness vs Readiness Probes
Liveness probe"Is this container still alive?" If it fails repeatedly, Kubernetes kills and restarts the container. Use for detecting deadlocks/hangs.
Readiness probe"Is this container ready to receive traffic right now?" If it fails, the Pod is removed from the Service's routing (but NOT restarted) — used for things like "still warming up a cache" or "temporarily overloaded, back off."
Analogy: Liveness is a pulse check — no pulse, you get resuscitated (restarted). Readiness is asking a waiter "can you take an order right now?" — if they're mid-carrying a heavy tray, you don't fire them, you just don't send them a new table until they say they're ready again.
Common interview trap
The mistakeMaking the liveness probe check downstream dependencies (like "can I reach the database?"). If the database blips, every Pod's liveness probe fails simultaneously, and Kubernetes restarts your entire fleet — making a downstream outage far worse. Liveness should only check "is this process itself stuck," not external dependencies.
Horizontal Pod Autoscaler (HPA)

Define: Automatically adjusts the number of Pod replicas based on observed metrics (CPU, memory, or custom metrics like request queue depth) against a target threshold. Why it exists: Same reasoning as an AWS Auto Scaling Group — match capacity to actual demand instead of provisioning for peak load 24/7.

Say this in interview

"The distinction I always draw is liveness answers 'should this be restarted,' readiness answers 'should this receive traffic right now' — and the classic bug is conflating them, where a liveness probe checking a downstream dependency turns one outage into a fleet-wide restart storm."

Networking — Ingress & Service Types

How traffic actually gets from outside the cluster to a Pod
Service types
ClusterIP (default)Only reachable from inside the cluster — for internal service-to-service communication.
NodePortOpens a specific port on every node — simple but clunky, rarely used directly in production.
LoadBalancerProvisions an actual cloud load balancer (an AWS ALB/NLB, an Azure LB) in front of the Service — the standard way to expose one service externally on a cloud provider.
Ingress

Define: A single entry point that routes external HTTP(S) traffic to multiple Services based on hostname or path — conceptually the Kubernetes-native equivalent of an ALB doing path-based routing, or an API Gateway sitting in front of several backend services.

Analogy: If each Service with its own LoadBalancer is like giving every department in a building its own separate street entrance, Ingress is one lobby with a directory board routing visitors to the right department based on who they're asking for — one entry point, many destinations, instead of N separate cloud load balancers costing N times as much.
Trade-off
Use Ingress whenMultiple services need external HTTP access — cheaper and more manageable than one LoadBalancer per service.
Use LoadBalancer directly whenExposing a single non-HTTP service (e.g. raw TCP), where Ingress's HTTP-routing features don't apply.
Say this in interview

"I default to Ingress over one LoadBalancer per Service, for the same reason I'd default to an API Gateway over exposing every microservice individually — one managed entry point, path/host-based routing, and it avoids paying for and managing N separate cloud load balancers."

ConfigMaps & Secrets

Externalizing configuration from the container image — same instinct as appsettings.json layering
ConfigMapNon-sensitive configuration (feature flags, URLs, log levels) injected into Pods as environment variables or mounted files — decouples config from the image so you don't rebuild a container just to change a setting.
SecretSame mechanism, for sensitive data (API keys, DB passwords, certs) — base64-encoded at rest by default, which is encoding, not encryption. Real protection requires enabling encryption-at-rest for etcd and/or integrating an external secrets manager (AWS Secrets Manager, HashiCorp Vault).
Common interview trap
The misconceptionAssuming Kubernetes Secrets are encrypted by default just because they're a separate object type from ConfigMaps — they're base64-encoded, which anyone with API access can trivially decode. This is one of the most common false assumptions candidates state confidently and get caught on.
Say this in interview

"I'm careful not to overstate what a Kubernetes Secret protects — out of the box it's base64 encoding, not encryption, so for anything genuinely sensitive I'd want etcd encryption-at-rest enabled and ideally integration with an external secrets manager, not just relying on the Secret object type alone."

Deployment Strategies

Rolling update, blue-green, and canary — same trade-off spectrum as any deployment strategy
Rolling Update (Deployment's default)Gradually replaces old Pods with new ones, a few at a time, keeping the service available throughout. Simple, built in, no extra infrastructure — but a bad new version is exposed to some real traffic before you notice.
Blue-GreenRun the full new version alongside the full old version, then switch traffic all at once. Instant rollback (just switch back), but requires double the infrastructure temporarily.
CanaryRoute a small percentage of real traffic (5%, 10%) to the new version first, watch metrics, then gradually increase. Lowest blast radius for a bad release, but needs more sophisticated traffic-splitting (often a service mesh like Istio, or Ingress-level weighting) and more operational maturity to run well.
Say this in interview

"I'd pick based on blast radius tolerance and infrastructure maturity — rolling update as the sensible default for most services, blue-green when instant rollback matters more than infrastructure cost, and canary when the team already has the traffic-splitting tooling and wants the smallest possible exposure for a risky release."

StatefulSets

For workloads where Pod identity and storage actually need to persist

Define: Unlike a Deployment's interchangeable, identical Pods, a StatefulSet gives each Pod a stable, unique identity (predictable names like db-0, db-1) and a persistent volume that follows that specific Pod even if it's rescheduled — used for databases, message brokers, and anything where "which specific instance" matters.

Analogy: A Deployment's Pods are like disposable coffee cups — any cup works, grab a new one if one breaks. A StatefulSet's Pods are like numbered lockers in a gym — locker #7 is always locker #7, keeps the same contents, even if you leave and come back later.
Trade-off
Use whenRunning stateful workloads directly in-cluster — databases, Kafka/Zookeeper, anything needing stable network identity and per-instance persistent storage.
Reality checkMost teams prefer a managed external database (RDS, managed Kafka) over running stateful workloads inside Kubernetes themselves — StatefulSets solve a real problem, but running your own database in K8s adds significant operational burden most teams choose to avoid when a managed alternative exists.
Say this in interview

"StatefulSets exist for real reasons — stable identity and per-Pod storage — but in practice, I'd lean toward a managed database service over running one inside Kubernetes via a StatefulSet, unless there's a specific reason (cost, portability, or an existing platform standard) pushing toward running it in-cluster."

Rapid Recall Drill

Why does a Service exist if Pods already have IPs?
Pod IPs are ephemeral and change constantly — a Service gives callers a stable identity that doesn't change as Pods are recreated underneath it.
Liveness probe checks the database and it's temporarily down — what happens?
Every Pod's liveness probe fails simultaneously, and Kubernetes restarts the entire fleet — a fleet-wide restart storm caused by a downstream dependency, which is why liveness should never check external dependencies.
Are Kubernetes Secrets encrypted by default?
No — base64-encoded, which is trivially reversible. Real protection needs etcd encryption-at-rest and/or an external secrets manager.
Cheapest way to expose 5 HTTP services externally?
One Ingress routing by host/path, instead of 5 separate LoadBalancer Services (5x the cloud load balancer cost).
Lowest blast-radius deployment strategy for a risky release?
Canary — small percentage of real traffic first, watch metrics, then ramp up.
When do you reach for a StatefulSet instead of a Deployment?
When Pod identity and per-instance persistent storage matter — e.g. running a database or Kafka directly in-cluster — though a managed external service is often preferred in practice.
