Change Detection — Default vs OnPush

How Angular decides what to re-render, and how you make it stop over-checking

Define: Default strategy checks every component in the tree on every change-detection cycle (triggered by any event, timer, or async operation via Zone.js). OnPush only re-checks a component when its @Input() reference changes, an event originates from inside it, or an observable it's subscribed to (via the async pipe) emits.

Analogy: Default is a teacher checking every single student's work after every single question, all 40 students, all the time. OnPush is a teacher who only checks a student's work when that student raises their hand — far less wasted effort, but you must remember to actually raise your hand (emit a new reference) or the teacher will assume nothing changed.
Why it exists

In a large app, Default's "check everything, every time" doesn't scale — a click anywhere can trigger a full-tree check. OnPush is an opt-in performance contract: you promise the component only needs re-checking on specific, well-defined triggers.

Trade-off
Use OnPush whenThe component's inputs are immutable/replaced-not-mutated, and you're building for scale or performance-sensitive views (large lists, dashboards).
The classic trapMutating an object/array in place (e.g. this.items.push(x)) doesn't change the reference, so OnPush won't detect it — you must reassign (this.items = [...this.items, x]).
Say this in interview

"OnPush is a performance contract you're making with Angular — the component promises to only care about reference changes. The bug I always check for first when 'the UI isn't updating' with OnPush is a mutation instead of a reassignment somewhere upstream."

Signals

Fine-grained reactivity that tracks exactly what changed, not "did anything, maybe, somewhere"

Define: A Signal is a reactive value container (signal(0)) that automatically tracks which parts of the template/computed values depend on it, and updates only those — no Zone.js, no tree-wide checks needed.

Analogy: Zone.js-based change detection is like a fire alarm that goes off building-wide for any smoke anywhere. Signals are individual smoke detectors wired directly to the sprinkler above them — only the affected area reacts, because the dependency is explicit and direct.
Why it exists

Zone.js change detection can't know exactly what changed — it has to re-check broadly. Signals make dependencies explicit at the point of use, so Angular knows precisely which DOM bindings need updating, without needing OnPush's manual discipline about references.

Trade-off
Use whenNew code, especially performance-sensitive UI — Signals give OnPush-like precision without OnPush's mutation gotchas.
Migration realityExisting large RxJS-based codebases can't switch overnight — Signals and RxJS interop (toSignal/toObservable) exist specifically for gradual migration, not a rip-and-replace.
Say this in interview

"Signals solve the same problem OnPush does — avoiding tree-wide checks — but by making dependencies explicit rather than relying on the developer remembering reference-immutability discipline. I'd treat adopting them in an existing codebase as incremental, using the RxJS interop rather than a big-bang rewrite."

RxJS Operators

The handful of operators that show up in nearly every real Angular app
switchMap vs mergeMap vs concatMap vs exhaustMap

Analogy — same "search box calling an API" scenario, four different behaviors:

  • switchMap — a search-as-you-type box: cancels the previous in-flight request the moment a new keystroke fires. Use for: typeaheads, "only the latest result matters."
  • mergeMap — firing multiple file uploads in parallel, all running concurrently, no cancellation. Use for: independent parallel operations.
  • concatMap — a queue of emails that must send in order, one fully completing before the next starts. Use for: order-sensitive sequential operations.
  • exhaustMap — a login button that ignores further clicks until the current login attempt finishes. Use for: ignoring new triggers while one is in flight.
Why it exists

Async operations that themselves return observables (an HTTP call inside a stream) need a defined strategy for handling overlap — these four operators are really just four different overlap-handling policies, not four unrelated tools.

Say this in interview

"I pick these four by asking one question: what should happen if a new trigger fires while the last one's still in flight? Cancel and restart → switchMap. Run both → mergeMap. Queue it → concatMap. Ignore it → exhaustMap. That framing is more useful in an interview than memorizing definitions."

Trade-off / common bug
Memory leaksManual .subscribe() calls that are never unsubscribed leak memory as components are destroyed and recreated. Fix: the async pipe in templates (auto-unsubscribes), or takeUntilDestroyed() in newer Angular versions.

NgModules vs Standalone Components

Angular removing a layer of ceremony that was mostly bookkeeping

Define: NgModules (@NgModule) used to be mandatory — every component had to be declared in exactly one module, which controlled what it could inject and what could use it. Standalone components declare their own imports directly, no module required.

Analogy: NgModules are like needing a company org chart updated every time you hire someone, just so they know who they can talk to. Standalone components are like each new hire just being handed their own contact list directly — less bureaucracy, same eventual connections.
Trade-off
Standalone benefitSimpler mental model, better tree-shaking, less boilerplate for new projects — this is the default direction for new Angular apps now.
Migration realityLarge existing NgModule-based codebases migrate incrementally — Angular supports both models interoperating, so it's not forced all-at-once.
Say this in interview

"Standalone components remove bookkeeping that NgModules required without adding new capability — it's the same DI and component model underneath, just less ceremony to wire up. For an existing large app, I'd migrate module-by-module rather than treat it as a rewrite."

State Management

BehaviorSubject vs NgRx vs Signals — matching the tool to the actual complexity
BehaviorSubject serviceSimplest option — a service holding a BehaviorSubject, exposed as an Observable. Good for small-to-medium apps with straightforward shared state.
When it breaks downOnce you need time-travel debugging, complex derived state, or many features touching the same state with unclear update order, plain BehaviorSubjects get tangled fast.
NgRxRedux pattern — single store, actions, reducers, effects for side-effects. Strong for large teams needing strict, traceable, testable state transitions.
NgRx costSignificant boilerplate (actions, reducers, selectors, effects per feature) — genuinely overkill for a small app; teams often over-adopt it too early.
Signals for stateNewer, lighter-weight option for local/moderate shared state — less ceremony than NgRx, more structure than a raw BehaviorSubject.
Still maturingSignal-based global state patterns are newer and less battle-tested at NgRx's scale for very large, multi-team apps.
Say this in interview

"I pick state management by actual complexity, not by default reaching for NgRx. A BehaviorSubject-backed service is enough for most feature-level shared state. I'd bring in NgRx specifically when I need traceable, testable state transitions across a large team — not because 'that's what Angular apps use.'"

Rapid Recall Drill

UI not updating with OnPush — first thing to check?
Is state being mutated in place instead of reassigned? OnPush only reacts to new references.
Typeahead search — which RxJS operator?
switchMap — cancel the stale request when a new one comes in.
Why do Signals not need Zone.js?
Dependencies are tracked explicitly at the point of use, so Angular knows exactly what to update, rather than needing to guess by re-checking broadly.
When is NgRx overkill?
Small-to-medium apps without complex cross-feature state — the boilerplate cost outweighs the traceability benefit.
