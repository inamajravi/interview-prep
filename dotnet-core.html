DI Lifetimes

Singleton, Scoped, Transient — and the bug each one causes when misused

Define: Three lifetimes control how long the container keeps an instance alive: Singleton (one instance, app lifetime), Scoped (one instance per HTTP request), Transient (new instance every time it's injected).

Analogy: Singleton is the office's shared printer — everyone uses the same one. Scoped is a meeting room booked for one meeting — fresh per meeting, shared by everyone inside that meeting. Transient is a disposable coffee cup — brand new every single time, never shared.
Why it exists

Not everything should be shared (a DbContext per-request avoids cross-request data leaking), and not everything should be recreated constantly (an expensive-to-build config object should live for the app's whole life). Lifetimes let you match object lifespan to its actual use.

My real example

In the sanctions API, the loaded watchlist data (OFAC/UN lists) is effectively singleton-scoped — expensive to parse, safe to share read-only across requests. The per-request screening context (which name is being checked, which thresholds apply) is scoped — fresh per screening call, never bleeding into the next request.

Trade-off — the classic bug
Captive dependencyInjecting a Scoped service into a Singleton captures that scoped instance for the app's entire lifetime — silently breaking per-request isolation. This is the single most common DI interview trap question.
FixNever inject Scoped/Transient into Singleton directly — inject IServiceScopeFactory and create a scope manually when the Singleton needs scoped work done.
Say this in interview

"The lifetime bug I watch for most is a captive dependency — a Scoped service accidentally getting pinned to Singleton lifetime because it was injected into one. I always check the lifetime of everything a Singleton depends on, not just the Singleton itself."

Middleware Pipeline

Chain of Responsibility, applied to every HTTP request

Define: Each request flows through an ordered chain of middleware components, each able to inspect/modify the request, call the next middleware, and inspect/modify the response on the way back out.

Analogy: Airport security checkpoints — ID check, bag scan, metal detector, boarding gate — each in a fixed order, each able to stop you from proceeding, and you pass back through some of them (in reverse) on the way out.
Why order matters

Middleware order is not cosmetic — it's the whole design. UseExceptionHandler must be first (to catch everything downstream), UseAuthentication must come before UseAuthorization (you can't authorize an unauthenticated user), and UseRouting must come before endpoint-specific middleware that needs route data.

My real example

In the FastAPI sanctions project the same shape exists via dependency injection + middleware-equivalent hooks — request validation happens before the matching engine ever runs, and error handling wraps the whole call so a malformed name doesn't crash the process, it returns a clean 400.

Trade-off
Use middleware forCross-cutting concerns: logging, auth, exception handling, CORS, response compression.
Don't put in middlewareBusiness logic — middleware should never know about your domain (e.g. "is this a valid sanctions match"); that belongs in a service, not the pipeline.
Say this in interview

"Middleware order is the interview trap — I always reason about it as a Chain of Responsibility, and the two orderings I double check every time are exception handling being outermost, and authentication strictly before authorization."

Async/Await Internals

It's not multithreading — it's not blocking a thread while waiting

Define: async/await lets a thread be released back to the thread pool while an I/O operation (DB call, HTTP call, file read) is in flight, instead of blocking that thread doing nothing. When the I/O completes, a thread (not necessarily the same one) resumes the method from where it left off.

Analogy: A waiter at a restaurant doesn't stand at your table staring at the kitchen while your food cooks — they take other tables' orders and come back when your food's ready. The waiter is the thread; standing idle for 20 minutes would be a blocking call; serving other tables in the meantime is async.
Why it exists

Without async, a web server handling 1,000 concurrent slow DB calls would need 1,000 blocked threads sitting idle — expensive and doesn't scale. Async lets a small thread pool serve thousands of concurrent I/O-bound requests, since threads are freed during the wait.

Trade-off / common trap
async voidNever use except for event handlers — exceptions in an async void method can't be caught by the caller and will crash the process.
Sync-over-asyncCalling .Result or .Wait() on a Task can deadlock in contexts with a SynchronizationContext (classic ASP.NET, some UI apps) — always await all the way up.
Say this in interview

"Async/await is about thread efficiency for I/O-bound work, not parallelism — it doesn't make a single operation faster, it lets one thread serve many concurrent waiting operations. The bug I'm most careful about is sync-over-async — calling .Result on a Task — because it can deadlock in the wrong context."

▸ Deep dive: what the compiler actually generates

An async method compiles into a state machine — each await is a point where execution can suspend and later resume, implemented via a generated class implementing IAsyncStateMachine. ConfigureAwait(false) tells the continuation it doesn't need to resume on the original context (e.g. UI thread) — a common library-code optimization to avoid unnecessary context switches.

EF Core

Change tracking, and the two bugs that account for most EF Core interview questions

Define: EF Core is an ORM — DbContext tracks entity state (Added/Modified/Deleted/Unchanged) and translates LINQ into SQL, generating the right INSERT/UPDATE/DELETE on SaveChanges().

The N+1 problem

Looping over a list and accessing a lazy-loaded navigation property per iteration fires one query per row instead of one query total. Fix: .Include() for eager loading, or explicit projection with .Select() to only pull needed columns.

Tracking vs No-Tracking

By default EF Core tracks every entity it loads (to detect changes for SaveChanges) — for read-only queries (like a screening lookup that never modifies data), .AsNoTracking() avoids the tracking overhead entirely and is a real, measurable performance win on hot read paths.

Trade-off
Use EF Core whenStandard CRUD, rapid development, and the abstraction's overhead is acceptable.
Drop to Dapper/raw SQL whenA query is performance-critical and EF's generated SQL is inefficient — you don't need to abandon EF app-wide, just for that hot path.
Say this in interview

"The two EF Core habits I always apply: AsNoTracking on read-only paths, and watching for N+1 by checking generated SQL (via logging) whenever a loop touches a navigation property. Most EF Core performance issues I've seen trace back to one of those two."

Minimal APIs

Same HTTP pipeline, far less ceremony for small, focused services

Define: Define endpoints directly (app.MapGet("/screen", handler)) without a full Controller class — same routing/DI/middleware pipeline underneath, just less boilerplate.

Analogy: Controllers are a formal restaurant with a fixed menu structure (starters, mains, desserts as separate classes). Minimal APIs are a food truck — same food, same kitchen, far less ceremony to order.
Trade-off
Use forMicroservices with a small number of focused endpoints, or when you want the leanest possible startup/footprint.
Avoid forLarge APIs with many endpoints — Controllers' organization (grouping by resource, filters, model binding conventions) scales better for big surface areas.
Say this in interview

"Minimal APIs are a good fit for small, focused microservices where the ceremony of full MVC controllers isn't earning its keep — but for a large API surface, I'd still reach for Controllers because the organizational conventions pay off at scale."

Rapid Recall Drill

What breaks if you inject a Scoped service into a Singleton?
Captive dependency — the scoped instance gets pinned to the singleton's lifetime, silently breaking per-request isolation.
Why does async/await help throughput?
It frees the thread during I/O wait instead of blocking it, so a small thread pool can serve many concurrent I/O-bound requests.
Fastest fix for N+1 in EF Core?
.Include() for eager loading, or restructure the query to project only what's needed instead of looping and lazy-loading.
Why AsNoTracking on reads?
Skips EF's change-tracking overhead for entities you'll never modify — real performance win on read-heavy paths.
