Blueprint 10 / Final Prep

Mock Interview Drill

Cross-domain, rapid-fire, no reference material. Cover the answer, read the question out loud, answer in the Define → Why → Example → Trade-off shape, then check yourself. Do this the night before and the morning of.

TARGET: under 45 seconds per answer, out loud
01 Warm-up (Your Story) 02 Architecture 03 .NET / Spring 04 Angular 05 Cloud 06 Curveballs 07 Closing Move

Warm-up — Your Story

These will come first. Have them loaded and ready, no hesitation.
Walk me through a project you're proud of.
Sanctions screening API — architect/reviewer with AI implementing. Lead with the domain complexity (weighted multi-algorithm matching across multiple watchlist sources), not the tools.
Tell me about a time you disagreed with a technical decision.
Pull a real one from your own experience — interviewers can tell a generic STAR answer apart from a specific one. Anchor it in an architecture trade-off you actually navigated.
How do you approach mentoring or leading a team technically?
Frame around the architect-reviewer pattern: setting constraints upfront, reviewing critically, catching issues before they compound — same skill whether the contributor is a junior dev or an AI tool.

Architecture — Rapid Fire

When would you NOT use microservices?
Small team, small domain, early-stage product — pay the distributed-systems tax only once you need the benefit of independent deployment/scaling.
Does CQRS require two databases?
No — can be two models against one database; two stores is an optimization added only when read/write scaling genuinely diverges.
How do you handle a transaction spanning two microservices?
Saga pattern — sequence of local transactions with compensating actions on failure, not a distributed transaction.
What's a Bounded Context, and why does it matter for microservices?
An explicit boundary where a domain term has one specific meaning — a natural seam for drawing microservice boundaries around real business capabilities.
Choreography vs orchestration for a multi-step workflow — which do you default to?
Orchestration once steps exceed 3-4, for observability; choreography for simpler, fully decoupled reactions.

.NET / Spring — Rapid Fire

What breaks if you inject a Scoped service into a Singleton?
Captive dependency — scoped instance gets pinned to singleton lifetime, breaking per-request isolation.
Why doesn't async/await make one operation faster?
It frees the thread during I/O wait rather than speeding up the I/O itself — throughput gain, not latency gain, for a single call.
Fastest fix for N+1 queries?
Eager loading (.Include() in EF Core, JOIN FETCH / @EntityGraph in JPA) instead of lazy-loading inside a loop.
Default Spring bean scope, and why does it surprise .NET developers?
Singleton — unlike .NET where scoped-per-request is the common default for web-facing services, so mutable state on a default Spring bean is shared across every request.
What is @Transactional doing under the hood?
AOP proxy wrapping the method to start/commit/rollback a transaction — declarative version of manually wrapping code in a transaction block.

Angular — Rapid Fire

OnPush component isn't updating — first thing you check?
Is state mutated in place instead of reassigned? OnPush only reacts to reference changes.
Typeahead search box — which RxJS operator, and why?
switchMap — cancels the stale in-flight request when a new keystroke arrives; only the latest result matters.
When is NgRx overkill?
Small-to-medium apps without complex cross-feature state — boilerplate cost outweighs traceability benefit; a BehaviorSubject-backed service is often enough.
Why do Signals not need Zone.js-style tree-wide checks?
Dependencies are tracked explicitly at point of use, so Angular knows exactly what needs updating.

Cloud (AWS/Azure) — Rapid Fire

Prerequisite for horizontal scaling to actually work?
Statelessness — session state externalized, not held in-process.
When is serverless (Lambda/Functions) the wrong choice?
Sustained high, predictable throughput (cost), or latency-sensitive workloads (cold starts).
ALB vs NLB — deciding question?
Do I need to route based on HTTP content? Yes → ALB (Layer 7). No, just raw fast TCP → NLB (Layer 4).
One event, three independent downstream reactions — which pattern?
Fan-out: SNS to multiple SQS queues (AWS) or a Service Bus Topic with multiple subscriptions (Azure).
Why put an API gateway (APIM) in front of microservices?
Centralize auth, rate limiting, versioning — avoid duplicating cross-cutting concerns per service.

Curveballs — Questions Designed to Catch Overconfidence

These test judgment, not recall. The "I don't know, here's how I'd find out" answer is often the right one.
"You said you used AI to build that API — so what did YOU actually do?"
Be specific and concrete: the architectural decisions (algorithm choice and weighting), the interface contracts, the bugs you personally diagnosed, the improvements you proactively identified. Never get defensive — restate the architect/reviewer framing calmly.
"How would you handle a disagreement with a senior engineer who wants a different architecture?"
Show you'd engage with their reasoning first, identify the actual trade-off being debated, and escalate to data/prototyping rather than authority if it's not resolved — avoid a pure "I'd just explain why I'm right" answer.
A question on a tool/topic you genuinely don't know well
Use your partial-knowledge rule: "I know [related/adjacent thing] well — on this specific piece my understanding is more surface-level, so I'd want to validate it rather than guess." This is a stronger answer than bluffing.
"Walk me through how you'd design X" (unfamiliar system, on the spot)
Narrate your process out loud: clarify requirements/scale first, identify the core domain entities, propose a first-pass architecture, then explicitly name the trade-offs of your choice. The process matters more than landing on a "perfect" answer.

Closing Move

The last impression, and the one thing worth repeating if you get the chance
If asked "anything else you'd like to add?"

"The thread running through everything I've talked about is that the Tech Lead job is fundamentally about architectural judgment and review discipline — whether that's reviewing a teammate's PR, weighing microservices trade-offs, or directing AI-generated code toward a specific design. I've been applying that same discipline daily, and I'd bring it directly into this role."

Final checklist before you walk in
  • Can you say your AI/sanctions-API story in under 30 seconds, unprompted, calmly?
  • Do you have one specific trade-off story ready for Architecture, .NET/Spring, Angular, and Cloud each?
  • Have you rehearsed the "I don't know X well, but here's Y" phrasing out loud at least once?
  • Do you have 2-3 thoughtful questions ready to ask them about the role/team?
