Your Core Story — The Sanctions Screening API

Say this early, unprompted, if the interview gives you an opening

The one-paragraph version: "I built a production-grade sanctions screening REST API from scratch using GitHub Copilot and Claude, with FastAPI and Python, acting as the architect and reviewer while AI handled implementation. It grew to four endpoints — /screen, /forecast, /explain, and a health check — with weighted multi-algorithm matching combining fuzzy, phonetic, and Levenshtein distance scoring, multi-list support across OFAC SDN, OFAC Non-SDN, and UN sources, and a config-driven, layered architecture. I independently debugged issues and proactively identified architectural improvements along the way."

Why this story is strong

It's not "I used ChatGPT to write some code." It has: a real domain with genuine complexity (compliance/sanctions matching), a specific architectural decision you can defend (why multiple weighted algorithms instead of one), a real technical challenge (multi-source config-driven design), and evidence of judgment (you debugged and improved the AI's output, not just accepted it).

Trade-off — the honest caveat

Frame it precisely: "I acted as architect and reviewer — AI wrote the implementation, I made the design decisions, caught the issues, and drove the direction." This is more credible than either extreme (claiming to have hand-typed everything, or implying you just accepted AI output uncritically). Interviewers respect the accurate framing more than an inflated one.

Say this in interview

"The most senior part of the job wasn't writing code — it was making the architectural calls: which matching algorithms to combine and how to weight them, how to structure config so a new watchlist source doesn't require touching the matching engine, and catching when the AI's first pass missed an edge case. That's the same judgment a Tech Lead role needs, just applied with a different tool in hand."

What "Agentic AI" Actually Means

The difference between a chatbot and an agent is loops and tools

Define: A chatbot answers one question and stops. An "agentic" system can plan a multi-step task, call tools/functions to take real actions (read a file, run code, call an API), observe the result, and decide the next step — looping until the goal is done, without a human in the loop for every step.

Analogy: A chatbot is a knowledgeable friend you call for advice — they tell you what to do, but you go do it. An agent is a friend who says "give me the keys, I'll go handle it," checks in only if something unexpected comes up, and comes back when it's done.
Why it exists

Real engineering tasks (build this API, fix this bug, refactor this module) aren't single-shot Q&A — they require exploring a codebase, making a change, running tests, seeing what broke, and iterating. Agentic tools like Copilot's agent mode or Claude Code close that loop instead of requiring a human to manually paste code back and forth at every step.

My real example

Building the sanctions API wasn't one prompt producing one file — it was an iterative loop: propose an architecture, have Copilot implement a piece, run it, hit an issue (a matching edge case, a config bug), diagnose it, and redirect the next iteration. That loop — direct, implement, observe, redirect — is the practical shape of working with an agentic tool, even without deep knowledge of the orchestration internals.

Trade-off
Where agents help mostWell-scoped, verifiable tasks (tests pass/fail, code compiles) where iteration is cheap.
Where human judgment stays essentialArchitectural decisions, security-sensitive code, ambiguous requirements — agents can drift confidently in a wrong direction without a reviewer checking in.

Prompt Engineering Patterns

The handful of habits that actually move the needle in practice
  • Be specific about constraints, not just goals. "Add fuzzy matching" is weak. "Add fuzzy matching using token-sort-ratio, weighted at 40% of the composite score, following the existing IMatcher interface" gets a usable result on the first pass.
  • Show, don't just tell — give an example of the shape you want. Pasting one existing matcher class as a pattern to follow produces far more consistent output than describing the pattern in prose.
  • Ask for the plan before the code, on non-trivial tasks. Having the AI outline its approach first lets you catch a wrong direction before dozens of lines of code are generated around it.
  • Iterate in small verifiable steps rather than one giant "build the whole feature" prompt — matches the same instinct as small, reviewable pull requests.
Say this in interview

"The habit that improved my results the most was asking for a plan before code on anything non-trivial — it turns a review into catching a wrong assumption in one paragraph, instead of catching it after 200 lines of generated code built on that assumption."

MCP — Model Context Protocol

A standard way for an AI model to discover and call external tools, instead of a bespoke integration per tool

Define: MCP is an open protocol that lets an AI assistant connect to external data sources and tools (a database, a file system, an API like Slack or GitHub) through a standard interface, rather than every AI product needing a custom-built integration for every tool.

Analogy: Before USB, every peripheral needed its own proprietary port and driver — a printer port, a mouse port, a scanner port. USB standardized the connection so any compliant device works with any compliant computer. MCP is doing the same thing for AI-to-tool connections — one protocol, many tools, many AI clients, instead of a custom integration per pair.
Why it exists

Without a standard, connecting an AI assistant to your company's internal ticketing system, database, and Slack each requires a separate, bespoke integration — N tools × M AI clients means N×M integrations. A shared protocol turns that into N+M: build one MCP server per tool, and any MCP-compatible client can use it.

Core pieces

MCP Server: exposes a tool/data source (e.g. "query this Postgres database") via the protocol. MCP Client: the AI application (e.g. Claude, an IDE) that discovers and calls what the server exposes. Tools: the specific actions/functions a server exposes for the model to call, with structured inputs and outputs.

Trade-off
Where it pays offAny org connecting AI assistants to multiple internal systems — the standardization avoids N×M integration cost.
Where it's overheadA single one-off integration for a single tool used by a single AI client — a direct API call may be simpler than standing up a full MCP server for it.
Say this in interview

"MCP is solving the same integration-explosion problem that standards like ODBC or USB solved in their domains — turning an N×M integration problem into N+M by standardizing how an AI model discovers and calls a tool. I'd frame it as infrastructure for connecting AI to systems, not a feature of any one AI product."

The Architect-Reviewer Role

This is the framing that maps your AI experience directly onto the Tech Lead job description

The core argument: A Tech Lead's job was never "write every line of code" — it's making architectural decisions, reviewing others' work for correctness and design quality, and catching issues before they compound. Working with AI as an "architect/reviewer" is the same skill, just applied to a different kind of contributor.

Concrete parallels to draw
  • Reviewing a junior dev's PR for missed edge cases ↔ reviewing Copilot's generated matching logic for missed edge cases
  • Setting architectural constraints for a team ↔ setting the interface contracts (IMatcher) and config-driven structure that AI-generated code had to follow
  • Catching a design smell before it ships ↔ catching that a naive single-algorithm matcher wouldn't handle transliteration variance, and redirecting toward the weighted multi-algorithm approach
Say this in interview

"I think the architect-reviewer skill set is exactly what transfers to a Tech Lead role, regardless of whether the 'contributor' is a person or an AI tool — the job is the same: set the direction, review critically, catch what's wrong before it compounds. If anything, working this way daily has sharpened my code review instincts, because I'm doing it constantly instead of only during scheduled PR reviews."
