Scaling — Vertical, Horizontal, Auto Scaling Groups

Bigger machine vs more machines, and letting AWS decide when

Define: Vertical scaling = bigger instance (more CPU/RAM on the same box). Horizontal scaling = more instances behind a load balancer. An Auto Scaling Group (ASG) automatically adds/removes instances based on metrics (CPU, request count, custom CloudWatch metrics).

Analogy: Vertical scaling is hiring a stronger single worker. Horizontal scaling is hiring more workers and splitting the work. An Auto Scaling Group is a manager who watches the line at checkout and calls in more cashiers only when the queue actually gets long, sending them home when it clears.
Why it exists

Vertical scaling has a hard ceiling (biggest instance type available) and a single point of failure. Horizontal scaling has no practical ceiling and survives individual instance failure — but requires the app to be stateless (or externalize state to a shared store), since you can't guarantee the same instance serves a user twice.

Trade-off
Vertical whenSimple, quick fix for a stateful legacy app not built for horizontal scaling; lower operational complexity.
Horizontal whenYou need real fault tolerance and elastic capacity — but only works cleanly if the app is stateless (session state externalized to Redis/DynamoDB, not in-process memory).
Say this in interview

"The real prerequisite for horizontal scaling is statelessness — if a service keeps session state in memory, adding instances just creates a 'sticky session' problem. I'd always check that before recommending horizontal scaling as the fix for a capacity problem."

Serverless — Lambda

No servers to manage — pay only for the compute time you actually use

Define: AWS Lambda runs your code in response to events (an HTTP request, a queue message, a file upload) without you provisioning or managing any server — AWS handles scaling, patching, and infrastructure entirely.

Analogy: Owning a car (a traditional server) means paying for it whether you drive it or not, and maintaining it yourself. Lambda is like a taxi — you pay only for the exact ride you take, and someone else maintains the vehicle. Great for occasional trips, expensive and awkward if you're commuting daily at scale.
Why it exists

Many workloads are bursty or infrequent (a nightly batch job, a webhook handler, an image-resize trigger) — running a dedicated server 24/7 for something that fires 50 times a day wastes money and operational effort. Lambda charges per invocation and per millisecond of execution.

Trade-off
Use whenEvent-driven, bursty, or infrequent workloads; short-lived tasks (under 15 min); you want zero infrastructure management.
Avoid whenConsistently high, predictable traffic (EC2/containers often cheaper at sustained scale); long-running processes; workloads sensitive to cold starts — the latency spike when a Lambda spins up from idle.
Say this in interview

"Lambda's cost model is the deciding factor for me — it's excellent for bursty or infrequent event-driven work, but at sustained high throughput, a always-on container or EC2 fleet is often cheaper than paying per-invocation. Cold start latency is the other real constraint I'd check against the specific use case's latency requirements."

ALB vs NLB

Layer 7 (application-aware) vs Layer 4 (raw speed) load balancing
ALB (Application Load Balancer)Operates at Layer 7 (HTTP/HTTPS) — can route based on URL path, host header, or headers. Supports WebSocket, SSL termination, and content-based routing (e.g. /api/* → one service, /admin/* → another).
NLB (Network Load Balancer)Operates at Layer 4 (TCP/UDP) — no awareness of HTTP content, just routes connections. Handles massive throughput with ultra-low latency, and supports static IP addresses.
Analogy: An ALB is a receptionist reading the actual content of your request ("I need Billing" → routes you to Billing). An NLB is a phone switchboard operator who just connects call to extension by number, without listening to what you say — much faster, but blind to content.
Trade-off
Use ALB whenMicroservices with path-based routing, need for HTTP-level features (cookies, headers, WebSocket).
Use NLB whenExtreme throughput/low-latency requirements, non-HTTP protocols, or needing a static IP for allowlisting.
Say this in interview

"The decision is really about whether I need to make routing decisions based on HTTP content — if yes, ALB; if I just need raw, extremely fast TCP-level load balancing without content awareness, NLB. Most typical microservices setups reach for ALB because path-based routing between services is exactly the Layer 7 feature they need."

SQS vs SNS

Queue (one consumer takes it) vs Topic (every subscriber gets a copy)
SQS (Simple Queue Service)A queue — a message is delivered to and processed by exactly one consumer (or one per consumer group), then removed. Good for work distribution and buffering/decoupling load.
SNS (Simple Notification Service)A pub/sub topic — every subscriber gets its own copy of the message. Good for fan-out: one event needs multiple independent reactions.
Analogy: SQS is a ticket queue at a deli counter — one ticket, one customer served, ticket gone. SNS is a fire alarm — everyone in the building hears it and reacts independently; the alarm doesn't "run out" after the first person responds.
The common combined pattern: SNS fan-out to SQS

Publish once to an SNS topic, with multiple SQS queues subscribed to it — each downstream service gets its own durable queue to process at its own pace, combining SNS's fan-out with SQS's durability and per-consumer buffering. This is the standard AWS pattern for "one event, several independent reactions, each reliably processed."

My real example

In the sanctions screening system, a "match found" event fanned out via SNS to separate SQS queues for notification, audit logging, and case management would let each downstream concern scale and fail independently — a notification service outage wouldn't block audit logging from happening.

Say this in interview

"SQS is for work distribution — one consumer per message. SNS is for fan-out — every subscriber gets a copy. When I need both durability and fan-out, I reach for the standard SNS-to-multiple-SQS pattern rather than picking one or the other."

Rapid Recall Drill

Prerequisite for horizontal scaling to work cleanly?
Statelessness — session state must be externalized, not held in-process.
When is Lambda the wrong choice?
Sustained high, predictable throughput — per-invocation billing gets expensive versus an always-on fleet; also long-running or latency-sensitive workloads due to cold starts.
ALB or NLB for path-based routing between microservices?
ALB — Layer 7, HTTP-aware routing.
One event needs 3 independent downstream reactions — SQS or SNS?
SNS (fan-out) — ideally SNS fanning out to separate SQS queues so each reaction is also durable and independently scalable.
