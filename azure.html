AKS — Azure Kubernetes Service

Managed Kubernetes — Azure runs the control plane, you run the workloads
AWS EKS ↔ Azure AKS

Define: AKS is Azure's managed Kubernetes offering — Azure operates and patches the Kubernetes control plane (API server, etcd, scheduler) for you, while you manage the worker nodes and deploy your containerized workloads onto them.

Analogy: Running your own Kubernetes cluster is like being both the airline and the airport authority — building runways, air traffic control, everything. AKS is booking a flight — the airline (Azure) handles the runway, air traffic control, and safety inspections; you just show up with your cargo (containers) to ship.
Why it exists

Kubernetes itself is powerful but operationally heavy — running your own control plane means you own etcd backups, API server upgrades, and certificate rotation. AKS removes that burden, letting teams focus on workloads (Deployments, Services, Ingress) rather than cluster operations.

Trade-off
Use whenYou genuinely need Kubernetes' flexibility — complex multi-service deployments, need for fine-grained orchestration control, multi-cloud portability is a real requirement.
Avoid whenA simpler PaaS (Azure App Service, Container Apps) meets the need — Kubernetes' operational complexity (networking, RBAC, autoscaling config) is real overhead most small-to-medium apps don't need.
Say this in interview

"I'd only reach for AKS once the app genuinely needs Kubernetes' orchestration flexibility — for most services, a simpler PaaS option gets you to production faster with far less operational surface area to own. Kubernetes is a powerful default that's frequently over-selected for problems that don't need it."

Azure Functions

Azure's serverless compute — same idea as Lambda, different triggers ecosystem
AWS Lambda ↔ Azure Functions

Define: Event-triggered, serverless compute — code runs in response to a trigger (HTTP request, queue message, timer, blob upload) without you managing a server, billed per execution.

Hosting plan trade-off — the interview-relevant detail
Consumption planTrue pay-per-execution, scales to zero — cheapest for infrequent workloads, but has the same cold-start latency issue as Lambda.
Premium / Dedicated planPre-warmed instances (no cold start), predictable cost, VNet integration — the fix when Consumption's cold starts are unacceptable, at higher baseline cost.
Say this in interview

"The plan choice is the real architectural decision with Functions — Consumption gives you true pay-per-use but with cold starts, Premium trades some of that cost efficiency for pre-warmed instances when latency matters. I'd pick based on the specific latency tolerance of the trigger, not default to Consumption everywhere."

Service Bus

Azure's enterprise messaging — combines what SQS and SNS do separately, plus more
AWS SQS + SNS ↔ Azure Service Bus (Queues + Topics)

Define: Service Bus offers both Queues (point-to-point, one consumer per message — like SQS) and Topics (pub/sub, multiple subscriptions each get a copy — like SNS's fan-out), unified in one service, plus enterprise features like message sessions (ordered processing per session key) and duplicate detection built in.

Analogy: Where AWS gives you a deli-ticket queue (SQS) and a separate fire alarm (SNS) as two different products, Service Bus is one building with both a ticket counter and an alarm system built into the same management console.
Trade-off
Use whenNeed for both patterns in one Azure-native service, or need Service Bus's extra enterprise features (sessions, transactions, dead-lettering) baked in.
Consider Event Grid instead whenNeed lightweight, high-scale event routing (closer to a pure event bus) rather than message queuing semantics — Service Bus is heavier-weight and more feature-rich than needed for simple event notification.
Say this in interview

"Service Bus unifies the SQS-style queue and SNS-style topic pattern I'd reach for on AWS into one service, with enterprise extras like ordered sessions and dead-lettering built in. For lightweight event routing without those enterprise needs, I'd consider Event Grid instead — it's a lighter-weight option, similar to the distinction between SNS and a heavier message broker."

APIM — API Management

A single front door for all your APIs — auth, rate limiting, versioning, in one place

Define: APIM sits in front of your backend APIs as a gateway, handling cross-cutting concerns centrally: authentication/authorization, rate limiting/throttling, request/response transformation, versioning, and analytics — without every backend service reimplementing them.

Analogy: A hotel's front desk is APIM — guests (API consumers) don't walk directly into the kitchen, housekeeping, or maintenance departments (individual backend services); they go through one front desk that checks identity, manages requests, and routes them appropriately.
Why it exists

Without a gateway, every microservice needs its own auth, rate limiting, and API-key management — duplicated logic and inconsistent enforcement. APIM centralizes it, and also decouples the public-facing API contract from internal service changes (you can version and transform without touching backend code).

Trade-off
Use whenMultiple APIs need consistent auth/throttling/versioning policy, or you're exposing internal services externally and need a controlled public contract.
Avoid whenA single small internal service with no need for a public contract or centralized policy — adds a network hop and operational component for no real benefit yet.
Say this in interview

"APIM's real value is centralizing cross-cutting API concerns — auth, throttling, versioning — so individual services don't each reimplement them inconsistently. I'd bring it in once I have multiple APIs needing that consistent policy, or when I'm exposing internal services externally and need a stable public contract decoupled from internal changes."

Rapid Recall Drill

AKS equivalent on AWS?
EKS — both are managed Kubernetes control planes.
Azure Functions plan for latency-sensitive triggers?
Premium/Dedicated — avoids Consumption plan's cold starts, at higher baseline cost.
Service Bus feature with no direct SQS/SNS equivalent?
Message sessions — ordered processing grouped by a session key, plus built-in duplicate detection.
Why put APIM in front of microservices?
Centralize auth, rate limiting, versioning, and transformation instead of duplicating them per service.
