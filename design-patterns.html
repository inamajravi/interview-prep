Design Patterns & SOLID

The vocabulary every architect interview leans on. Each pattern below: what problem it solves, a real analogy, and when NOT to reach for it — because knowing when to avoid a pattern is what separates senior from junior answers.

SOLID Principles

The five rules that make code changeable without fear
Analogy: A well-designed kitchen: each appliance does one job (SOLID's S), you can swap a hand mixer for a stand mixer without rewiring the kitchen (O and L), you don't wire the fridge into the toaster's circuit (I), and appliances plug into a standard socket, not directly into the wall's wiring (D).
S — Single Responsibility

Define: A class should have one reason to change. Why: When a class does 3 unrelated things, a change to one breaks the other two unexpectedly. My example: In the sanctions API, keeping fuzzy/phonetic/Levenshtein matchers as separate classes rather than one giant "Matcher god class" meant tuning one algorithm's weight never risked breaking another's logic. Trade-off: Taken too far, you get dozens of tiny classes and lose the big picture — a smell interviewers call "anemic over-decomposition."

O — Open/Closed

Define: Open for extension, closed for modification — add new behavior without editing existing, tested code. My example: Adding a 4th watchlist source (beyond OFAC SDN/Non-SDN/UN) should mean adding a new config entry and a new matcher implementation, not editing the core screening loop. Trade-off: Over-applying this upfront (speculative abstraction for extension points you'll never use) is wasted complexity — YAGNI cuts against blind Open/Closed application.

L — Liskov Substitution

Define: A subclass must be usable anywhere its base class is expected, without breaking correctness. Classic violation: a Square extending Rectangle that overrides SetWidth to also change height — breaks callers that assume width/height are independent. Trade-off: This is why "is-a" inheritance is often replaced with composition/interfaces in modern C# and Java — safer to satisfy LSP by design.

I — Interface Segregation

Define: Don't force a class to implement methods it doesn't need — many small interfaces beat one fat interface. My example: An IScreeningEngine interface with just Screen() is safer than one bloated IWatchlistService with Screen(), Ingest(), Audit(), Report() — a consumer that only screens shouldn't be forced to stub out Ingest(). Trade-off: Too many micro-interfaces make the codebase hard to navigate — balance matters.

D — Dependency Inversion

Define: Depend on abstractions, not concrete implementations — high-level modules shouldn't know the low-level details. My example: The screening service depends on an IMatcher interface, not a concrete FuzzyMatcher class directly — this is exactly what makes DI containers (services.AddScoped<IMatcher, FuzzyMatcher>) useful at all. Trade-off: None really — this is close to universally good, the only cost is one extra layer of indirection to read through.

Say this in interview

"SOLID isn't a checklist I apply mechanically — it's a set of smells I watch for. If I catch myself writing 'and' when describing what a class does, that's an SRP violation. If I'm forced to touch tested code to add a feature, that's an OCP violation. I try to name the smell, not just recite the letter."

Creational Patterns

Controlling how objects get built, so construction logic doesn't leak everywhere
Factory Method

Define: Delegate object creation to a method/class instead of calling new directly, so the caller doesn't need to know the concrete type. Analogy: Ordering "a coffee" at a counter — you don't specify the machine, beans, or grind; the barista (factory) decides based on what you ordered. My example: A MatcherFactory that returns the right matcher (fuzzy/phonetic/Levenshtein) based on config, so the screening loop never hardcodes a concrete matcher type. Trade-off: Adds a layer of indirection — overkill if you only ever have one implementation.

Builder

Define: Construct a complex object step-by-step instead of a constructor with 10 parameters. Analogy: Building a sandwich at Subway — you choose bread, then protein, then toppings, one decision at a time, rather than shouting all 10 choices at once. Trade-off: Unnecessary for objects with 2-3 simple fields — just use a constructor or object initializer.

Singleton

Define: Ensure exactly one instance of a class exists app-wide. My example: A shared, in-memory cache of loaded watchlist data (OFAC/UN) — you don't want every request reloading and re-parsing the sanctions list from disk. Trade-off: Notoriously overused — introduces global mutable state, makes unit testing harder (hidden dependency), and causes thread-safety bugs if not handled carefully. Most DI containers replace the need for manual Singletons — `AddSingleton` in .NET gives you this safely through the container instead of a static field.

Say this in interview

"I'm cautious with Singleton specifically — it's the pattern most likely to be overused. I'd rather let the DI container manage a single instance's lifetime than hand-roll a static singleton, because the container gives me the same benefit without the global-state testing pain."

Structural Patterns

Composing classes and objects into larger structures cleanly
Adapter

Define: Convert one interface into another the client expects, without changing either side. Analogy: A travel plug adapter — doesn't change the wall socket or your laptop charger, just bridges the mismatch. My example: If OFAC's API returns one JSON shape and UN's returns another, an adapter per source normalizes both into one internal WatchlistEntry shape before matching runs.

Decorator

Define: Add behavior to an object dynamically by wrapping it, without modifying its class. Analogy: Adding toppings to a pizza — each topping wraps the base without changing the dough recipe. My example: Wrapping the core matcher with a caching decorator or a logging decorator — same IMatcher interface, extra behavior layered on top. Trade-off: Many stacked decorators become hard to debug — you lose a single obvious place to set a breakpoint.

Facade

Define: Provide one simple interface over a complex subsystem. Analogy: A car's ignition switch — one turn of a key hides the complexity of starter motor, fuel injection, and spark plugs. My example: A single `ScreeningFacade.Screen(name)` hiding the orchestration across normalization → fuzzy match → phonetic match → weighted scoring → threshold check.

Proxy

Define: A stand-in that controls access to a real object — for lazy loading, caching, access control, or logging. My example: EF Core's lazy-loading proxies are a textbook example you already use daily — accessing a navigation property transparently triggers a DB query behind a proxy object.

Behavioural Patterns

How objects communicate and share responsibility for a behavior
Strategy

Define: Encapsulate interchangeable algorithms behind a common interface, selected at runtime. Analogy: Choosing a route on a GPS app — fastest, shortest, or avoid tolls — same destination, swappable strategy. My example: Fuzzy matching, phonetic matching, and Levenshtein distance are three interchangeable IMatchStrategy implementations combined with weights — the exact real pattern behind the sanctions API's multi-algorithm scoring. This is my strongest, most concrete real-world Strategy pattern story.

Observer

Define: One-to-many notification — subject notifies all subscribed observers when its state changes, without knowing who they are. Analogy: A YouTube channel notifying all subscribers on a new upload. My example: C#'s event/IObservable, or Angular's RxJS Subject — same pattern, different syntax. It's also the foundation event-driven architecture builds on at a larger scale.

Chain of Responsibility

Define: Pass a request along a chain of handlers until one handles it. Analogy: Tech support escalation — Tier 1 tries, escalates to Tier 2, then Tier 3. My example: ASP.NET Core middleware pipeline is exactly Chain of Responsibility — each middleware decides to handle, short-circuit, or pass along.

Template Method

Define: Define the skeleton of an algorithm in a base class, letting subclasses override specific steps. Analogy: A recipe card that says "prepare base, add filling, bake" — the filling step varies (apple vs cherry pie), the overall process doesn't. Trade-off: Relies on inheritance, which is less flexible than Strategy's composition — modern guidance often prefers Strategy over Template Method for this reason.

Say this in interview

"My clearest real pattern story is Strategy — the sanctions matching engine literally is three interchangeable scoring strategies combined with configurable weights. I'd lead with that rather than reciting textbook definitions, since it shows the pattern solving a real, specific problem."

Rapid Recall Drill

Name the pattern from the smell
"I have 5 nearly-identical algorithms selected by an if/else chain"
Strategy pattern — extract each branch into its own implementation of a shared interface.
"My constructor takes 9 parameters"
Builder pattern — or reconsider if the class is doing too much (SRP smell underneath).
"Two APIs return incompatible shapes for the same concept"
Adapter pattern.
"I need exactly one shared cache instance app-wide"
Singleton — but prefer registering it as AddSingleton in the DI container over a hand-rolled static.
"A request should be handled by whichever of several handlers is able to"
Chain of Responsibility — same shape as middleware pipelines.
