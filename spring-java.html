Dependency Injection & the IoC Container

Same DI you already do in .NET, different registration syntax
services.AddScoped<IFoo, Foo>() ↔ @Component / @Service + @Autowired

Define: Spring's ApplicationContext is the IoC container — exactly what .NET's built-in IServiceCollection / IServiceProvider is. You mark a class as a bean with @Component, @Service, @Repository, or @Controller (all specializations of @Component, just semantic labels), and Spring wires it into anything that asks for it.

Analogy: Same as .NET — it's the hotel concierge who hands you what you need without you going and building it yourself. The only real difference is how you check in: in .NET you explicitly register at the front desk (Program.cs); in Spring, most rooms self-register just by wearing the right badge (@Component) and being in a scanned package.
Why it exists

Identical reason to .NET: loose coupling, testability (swap real implementations for mocks), and centralizing object lifecycle instead of scattering new everywhere.

Lifetime mapping
.NETAddSingleton → one instance app-wide
AddScoped → one per HTTP request
AddTransient → new instance every injection
Spring@Scope("singleton") — default, one per container (not per-request!)
@Scope("request") → one per HTTP request
@Scope("prototype") → new instance every injection

⚠ Trap: Spring's default scope is singleton, not scoped-per-request like .NET's most common default. This is the #1 thing that surprises .NET developers moving to Spring — a naive mutable field on a default-scoped bean is shared across every single request.

Say this in interview

"Constructor injection, bean scopes, the container resolving the dependency graph — conceptually identical to what I do daily in .NET with the built-in DI container. The one thing I'd double check on a real Spring codebase is default bean scope, since it's singleton by default rather than per-request, which changes how you think about thread safety for mutable state."

▸ Deep dive: constructor vs field injection, @Qualifier

Constructor injection (preferred, same as .NET best practice): @Autowired on a constructor, or omit it entirely if there's only one constructor (Spring 4.3+ infers it). Field injection (@Autowired directly on a field) works but is discouraged — same reasoning as .NET: harder to test, hides required dependencies.

Multiple implementations: .NET's [FromKeyedServices]/named registration maps to Spring's @Qualifier("beanName") to disambiguate which implementation to inject when there's more than one bean of the same interface type.

Q: How does Spring know where to scan for @Component classes?
@ComponentScan (usually implicit via @SpringBootApplication on the main class) scans the package it's in and sub-packages — equivalent to explicitly registering assemblies/types in .NET, just convention-based instead of explicit.

Spring MVC / REST Controllers

Same controller pattern as ASP.NET Core, different attribute names
[ApiController] [HttpGet("id")] ↔ @RestController @GetMapping("/id")

Define: @RestController = [ApiController]. @RequestMapping/@GetMapping/@PostMapping = [Route]/[HttpGet]/[HttpPost]. @RequestBody = [FromBody]. @PathVariable = route parameter binding. @RequestParam = [FromQuery].

.NET 
[ApiController]
[Route("api/[controller]")]
public class ScreenController {
  [HttpPost]
  public IActionResult Screen(
    [FromBody] ScreenRequest req) {
    var result = _svc.Match(req);
    return Ok(result);
  }
}
Spring 
@RestController
@RequestMapping("/api/screen")
public class ScreenController {
  @PostMapping
  public ResponseEntity<Result> screen(
    @RequestBody ScreenRequest req) {
    var result = svc.match(req);
    return ResponseEntity.ok(result);
  }
}
Analogy: Same reception desk, different uniform. The controller's job — receive the request, validate the shape, hand off to a service, return a response — is identical. You're not learning a new job, just a new dress code for the same role.
Trade-off / things that differ
Validation.NET: Data Annotations + ModelState.
Spring: Bean Validation (@Valid + @NotNull/@Size from javax/jakarta.validation) — same idea, JSR-standard annotations instead of Microsoft's.
Global error handling.NET: exception-handling middleware.
Spring: @ControllerAdvice + @ExceptionHandler — a centralized handler class instead of pipeline middleware, same net effect.
Say this in interview

"The controller layer in Spring maps almost 1:1 to ASP.NET Core's — annotations instead of attributes, but the same separation of routing, model binding, and delegating to a service layer. The muscle memory transfers; what I'd want to get comfortable with fast is the exact validation and global exception-handling annotations Spring expects."

Spring Data JPA

Same idea as EF Core, but the repository is auto-generated for you
DbContext + LINQ ↔ JpaRepository + method names / JPQL

Define: JPA (Hibernate underneath, same relationship as EF Core sitting on top of raw ADO.NET) is Java's ORM standard. Spring Data JPA takes it further than EF Core by letting you define a repository interface with zero implementation — Spring generates the implementation at runtime from the method name itself.

.NET (EF Core) 
var user = await _db.Users
  .FirstOrDefaultAsync(u =>
    u.Email == email);
Spring Data JPA 
public interface UserRepo
  extends JpaRepository<User, Long> {
  Optional<User> findByEmail(
    String email);
}
// no implementation needed —
// Spring writes the query from
// the method name at runtime
Analogy: EF Core makes you write the sentence yourself (LINQ). Spring Data JPA is like a very literal assistant who writes the SQL just from reading the method's name out loud — findByEmailAndActiveTrue becomes a real query with zero code behind it, purely by naming convention.
Trade-off
Use derived queries whenSimple lookups by field(s) — fast to write, self-documenting.
Drop to @Query/JPQL whenQuery gets complex (joins, aggregations) — same judgment call as switching from LINQ method syntax to raw SQL/Dapper in .NET when LINQ gets unreadable.
Say this in interview

"It's the same ORM trade-off I already navigate in EF Core — lean on generated queries for simple CRUD, drop to explicit JPQL or native SQL when a query's complexity makes the generated approach unreadable or inefficient. The N+1 query problem is exactly the same failure mode as in EF Core's lazy loading — I'd handle it with JOIN FETCH the same way I'd use .Include() in EF Core."

▸ Deep dive: N+1 problem, transactions

N+1 queries: Just like EF Core lazy-loading a navigation property per row in a loop, JPA lazy-loads associations by default — fetching a list of Orders then accessing order.getCustomer() in a loop fires one query per order. Fix: JOIN FETCH in JPQL, or @EntityGraph, exactly parallel to EF Core's .Include().

Transactions: @Transactional on a service method = wrapping a method body in a using var tx = _db.Database.BeginTransaction() in EF Core, but declarative — Spring wraps the method via AOP proxy (see the AOP section below) instead of you writing the using block.

Spring Boot Autoconfiguration

Same idea as ASP.NET Core's minimal hosting model, but more "magic" by default

Define: @SpringBootApplication is really three annotations in one: @Configuration (this class can define beans), @ComponentScan (scan for @Components), and @EnableAutoConfiguration (guess sensible defaults based on what's on the classpath — e.g., see a JPA driver on the classpath, auto-configure a DataSource and EntityManager).

Analogy: .NET's minimal hosting (WebApplication.CreateBuilder) got closer to this over time, but Spring Boot pushes it further — it's like a smart hotel room that notices you packed running shoes and automatically sets up a gym pass, without you asking. Convenient, but you need to know it's happening so you're not surprised by config you never wrote.
Trade-off
BenefitMassive boilerplate reduction — a working REST + JPA app in one file plus one dependency list (starter POMs).
Cost"Magic" makes it harder to know exactly what got configured — same criticism .NET devs sometimes level at heavy convention-based frameworks. Debugging autoconfiguration requires --debug flag or the actuator's conditions report.
Say this in interview

"Autoconfiguration is convention-over-configuration taken further than ASP.NET Core typically goes — it infers a lot from what's on the classpath. I'd want to be comfortable reading the actual configuration report (via the Actuator or debug logging) so 'it just works' doesn't become a black box when something needs to be overridden."

▸ Deep dive: starter POMs, Actuator

Starter POMs: spring-boot-starter-web, spring-boot-starter-data-jpa etc. are curated dependency bundles — roughly the same idea as .NET's meta-packages (e.g. Microsoft.AspNetCore.App), just more granular and per-concern.

Actuator: maps to .NET's health check middleware + custom diagnostics — exposes /actuator/health, /actuator/metrics, etc. out of the box once you add the dependency.

Aspect-Oriented Programming (AOP)

Same problem middleware/decorators solve in .NET — cross-cutting concerns without cluttering business code
Middleware / Decorator pattern ↔ @Aspect + @Around/@Before/@After

Define: AOP lets you inject behavior (logging, security checks, transactions, retries) around method calls without modifying the method itself, by defining an "aspect" that matches a pattern of methods (a "pointcut") and runs code before/after/around them.

Analogy: Same idea as airport security — it doesn't care which specific flight you're on, it applies the same check to everyone passing through a certain gate. An aspect says "anything matching this pattern of methods gets this behavior wrapped around it," without each method needing to know security exists.
Why it exists

Same motivation as .NET middleware or a decorator wrapping a service: logging, auth checks, caching, and transaction handling are needed across many unrelated methods, and hardcoding them into every method violates single-responsibility and creates repetition.

Trade-off
Use whenA concern is truly cross-cutting (logging, security, transactions, caching) and repeating it manually would clutter every method.
Avoid whenOverused for core business logic — makes control flow hard to trace ("magic" happening outside the method you're reading), same criticism leveled at overusing decorators/middleware for things that should just be explicit code.
Say this in interview

"@Transactional is actually the clearest example — it's AOP under the hood, proxying the method to wrap it in a transaction, conceptually the same as middleware wrapping a request pipeline in ASP.NET Core. I'd apply the same discipline I use with middleware: reserve it for genuinely cross-cutting concerns, not business logic."

Spring Security

Same auth pipeline concepts as ASP.NET Core Identity/JWT middleware, different config style
app.UseAuthentication()/UseAuthorization() ↔ SecurityFilterChain bean

Define: Spring Security is a chain of servlet filters (conceptually the same as ASP.NET Core middleware pipeline) that intercept every request to authenticate and authorize it before it reaches your controller. You configure it via a SecurityFilterChain bean instead of ordering middleware calls.

Trade-off
SimilarJWT validation, role-based (@PreAuthorize("hasRole('ADMIN')") ↔ [Authorize(Roles="Admin")]), OAuth2/OIDC client support built in.
DifferentHistorically config-heavy/XML-flavored reputation; modern Spring Security (5+) uses a fluent Java DSL that reads much closer to .NET's fluent middleware config.
Say this in interview

"Conceptually it's the same filter/middleware chain intercepting requests before they hit a controller, with the same JWT and role-based authorization patterns I already use. The main adjustment is Spring Security's own DSL for wiring the filter chain — the concepts transfer, the exact configuration syntax is what I'd need reps on."

Profiles & Configuration

Same as appsettings.{Environment}.json + IOptions<T>
appsettings.Development.json ↔ application-dev.properties/yml

Define: application.properties/application.yml = appsettings.json. Spring Profiles (@Profile("dev"), activated via spring.profiles.active) = ASP.NET Core's ASPNETCORE_ENVIRONMENT-driven config layering. @ConfigurationProperties = strongly-typed IOptions<T> binding.

Say this in interview

"Profiles map directly to environment-based config layering in .NET — dev/staging/prod property files instead of appsettings.{Env}.json, and @ConfigurationProperties for strongly-typed binding is the same instinct as IOptions<T>, just YAML/properties instead of JSON."

Rapid Recall Drill — the full mapping table

Say the Spring term, then say the .NET equivalent, out loud
@Component / @Service / @Repository
Registering a class in IServiceCollection (services.AddScoped/Singleton/Transient) — semantic sub-types of the same idea as tagging a class for DI.
@Autowired
Constructor injection resolved by IServiceProvider — same mechanism, Spring can also do field/setter injection (discouraged, same as in .NET).
@RestController / @GetMapping
[ApiController] / [HttpGet] — controller + routing attributes.
@RequestBody / @PathVariable / @RequestParam
[FromBody] / route param / [FromQuery].
JpaRepository<T, ID>
DbSet<T> + a generic repository, but with auto-generated query methods from naming convention — EF Core doesn't do this part automatically.
@Transactional
using var tx = _db.Database.BeginTransaction(), but declarative via AOP proxy instead of explicit code.
@ControllerAdvice + @ExceptionHandler
Global exception-handling middleware.
@Value("${property}")
IConfiguration["Key"] — single value binding from config.
@Profile("dev")
ASPNETCORE_ENVIRONMENT-based appsettings.{Env}.json layering.
Bean scope: singleton (default) / prototype / request
AddSingleton / AddTransient / AddScoped — same three lifetimes, different default (Spring defaults to singleton, .NET has no single default — you must choose).
