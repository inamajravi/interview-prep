Big-O, the Way an Architect Uses It

Not proving bounds — reasoning about what breaks first at scale

Define: Big-O describes how an operation's cost grows as input size grows — not the exact speed, just the shape of the curve. O(1) constant, O(log n) logarithmic, O(n) linear, O(n log n) typical good sort, O(n²) nested loops over the same data.

Analogy: Finding a name in a phone book by flipping page by page is O(n) — double the book, double the worst-case time. Finding it by jumping to the middle and halving your search each time (because it's alphabetically sorted) is O(log n) — double the book, and you only need one extra flip. The structure of the data (sorted vs unsorted) is what changes the shape of the curve, not raw effort.
Why it matters at architect level

The question is never "what's the Big-O of X" in isolation — it's "this endpoint does an O(n²) operation over a list that's growing 10x a year, when does that become a production incident." An architect's job is spotting that curve before it bites, not reciting complexity classes.

My real example

In the sanctions API, comparing an input name against every entry in a watchlist naively is O(n) per screening call — fine at a few thousand entries, a real problem once a watchlist grows to hundreds of thousands. That's exactly the kind of thing that pushes you toward indexing/pre-filtering (e.g. a phonetic key lookup narrowing candidates before the expensive fuzzy comparison runs) rather than a brute-force full scan.

Say this in interview

"I care about Big-O as an early warning system, not a purity test — the real question I ask is what's the expected data growth over the next 1-2 years, and does today's algorithm's curve stay acceptable at that size. An O(n²) that's fine at 500 records is a production incident at 5 million."

Hash Map

The structure behind more system design answers than any other

Define: Key-value storage giving average O(1) lookup, insert, and delete by hashing the key to a bucket index. Trade: no ordering guarantee, and worst-case O(n) if hash collisions pile up in one bucket.

Analogy: A hash map is a coat check — you don't search every hook for your coat, the numbered ticket (hash) tells you exactly which hook to go to directly. A poorly-designed hash function is a coat check that sends everyone's coat to hook #7 — now you're searching through a pile at that one hook, back to O(n).
My real example

The idempotency check in the message-queue payment flow — "has this transaction ID already been processed?" — is a hash-map-shaped lookup at its core, whether backed by an in-memory cache or a database's indexed unique constraint. O(1) average lookup is exactly what makes that check cheap enough to do on every single message without becoming its own bottleneck.

Trade-off
Use whenYou need fast lookup by key and don't care about order — caches, dedup/idempotency checks, counting/frequency problems.
Avoid whenYou need sorted iteration or range queries — a hash map can't efficiently answer "give me everything between X and Y," a tree-based structure can.
Say this in interview

"Hash maps are my default for anything that's fundamentally 'look this up by key fast' — idempotency checks, caches, dedup. The moment a requirement includes ordering or range queries, that's my signal to reach for a tree-based structure instead."

Trees & Tries

When order and hierarchy matter, not just fast lookup
Balanced Binary Search Tree (B-Tree family)

Define: A sorted structure giving O(log n) search, insert, delete, while keeping data in order — enabling efficient range queries ("all entries between A and M"). My real example: This is essentially what a database index is — B-Trees are the standard structure behind SQL Server/Postgres indexes, which is exactly why an indexed WHERE clause with a range condition is fast, and why range queries on an un-indexed column are a full scan.

Trie (Prefix Tree)

Define: A tree where each path from the root spells out a prefix — built specifically for fast prefix matching. Analogy: A trie is exactly how a dictionary's thumb-tabs work — jump straight to "S," then "SA," narrowing character by character, instead of scanning every word. Use case: Autocomplete/typeahead search — retrieving all words starting with a given prefix in time proportional to the prefix length, not the size of the whole dataset.

Say this in interview

"When I hear 'need to search by prefix, fast, at scale' — autocomplete, typeahead — a trie is the structure purpose-built for that. When I hear 'need range queries on sorted data' — that's exactly what a database B-Tree index already gives me, so I don't need to reinvent it in application code."

Heaps & Priority Queues

Always knowing the "most important" item, without sorting everything

Define: A tree-shaped structure that keeps the minimum (or maximum) element accessible in O(1), with O(log n) insert/remove — without maintaining full sort order of everything else.

Analogy: A hospital ER's triage board doesn't fully sort every patient by severity — it just always knows who's most critical right now, and re-checks that as new patients arrive. That's a heap: cheap to always know "what's most urgent," without paying the cost of a full sort.
My real example / where this shows up in architecture

Any "process the most urgent thing next" system is a priority queue under the hood — a job scheduler processing high-priority tasks before low-priority ones, or a rate limiter's token bucket implementation. It's less about writing heap code yourself and more about recognizing when a library's PriorityQueue<T> (or a message broker's priority feature) is the right tool instead of hand-rolling sorting logic.

Say this in interview

"Whenever a requirement is 'always process the most urgent/smallest/largest item next, and new items keep arriving,' that's a priority queue problem — I'd reach for the built-in PriorityQueue rather than re-sorting a list on every insert, which would be needlessly O(n log n) repeated over and over."

Graphs

Anything with relationships between things — not just "social network" interview clichés

Define: Nodes connected by edges, representing relationships. BFS (breadth-first) explores level by level — good for shortest path in unweighted graphs. DFS (depth-first) explores as deep as possible before backtracking — good for detecting cycles, exploring all paths.

Analogy: BFS is how ripples spread across a pond from a dropped stone — everything at distance 1 first, then distance 2. DFS is exploring a maze by always taking the next available turn and only backtracking when you hit a dead end.
Where this actually shows up architecturally
  • Service dependency graphs — detecting circular dependencies between microservices is a cycle-detection problem (DFS-based).
  • Build/task ordering — resolving "run these steps in an order that respects dependencies" is topological sort, a graph algorithm — exactly what a CI/CD pipeline or a Terraform dependency graph resolves under the hood.
  • Recommendation/permission systems — "what can this role reach, transitively" is a graph traversal question.
Say this in interview

"I don't think of graphs as an interview-only topic — a microservices dependency map is a graph, and a circular dependency between services is literally a cycle in that graph, detectable the same way you'd detect a cycle in any directed graph."

LRU Cache — The Classic Combination Problem

Why this specific question gets asked so often: it combines two structures on purpose

Define: A cache with a fixed capacity that evicts the Least Recently Used item when full. Implemented by combining a hash map (O(1) lookup by key) with a doubly linked list (O(1) move-to-front on access, O(1) remove-from-tail on eviction) — neither structure alone gives you both O(1) lookup AND O(1) recency tracking.

Analogy: A stack of recently-worn clothes on a chair, plus a mental note of exactly where in the pile each item is. Wearing something moves it to the top of the pile (recency) instantly, but you also instantly know exactly where any specific item is in the pile (the hash map) rather than digging through it.
Why it's a favorite interview question

It's not really about either structure alone — it tests whether you recognize that combining two structures can satisfy two different requirements that neither one meets by itself. That combination instinct is the actual architect-level skill being probed, more than the specific data structure names.

Say this in interview

"LRU cache is really a lesson in combining structures — hash map alone gives me fast lookup but no cheap way to track recency; a linked list alone gives me cheap recency tracking but O(n) lookup. Combining them gets O(1) on both operations, which is the pattern I look for generally: what does neither single structure give me alone?"

Rapid Recall Drill

Need O(1) average lookup by key, don't care about order?
Hash map.
Need fast range queries on sorted data?
Balanced BST / B-Tree — this is what a database index actually is under the hood.
Need fast prefix search (autocomplete)?
Trie.
Need to always process the most urgent item next, as new items arrive?
Heap / priority queue.
Detecting a circular dependency between microservices?
Cycle detection on a directed graph (DFS-based).
Why does LRU cache need two structures, not one?
Hash map alone lacks cheap recency tracking; linked list alone lacks fast lookup — combining gives O(1) on both.
