Microservices Architecture

Small, independently deployable services, each owning one piece of the business

Define: An application built as a set of small, independently deployable services, each with its own database, each responsible for one business capability, talking to each other over the network (usually HTTP/REST or messaging).

Analogy: Think of a restaurant. A monolith is one chef doing everything — chopping, grilling, plating, billing — in one kitchen. If the grill breaks, the whole kitchen stops. Microservices is a restaurant with separate stations: a grill station, a salad station, a billing counter. Each has its own tools and staff. If the salad station has a bad day, the grill still serves steaks.
Why it exists

Monoliths become painful at scale: one team's bug can take down the whole app, one slow module blocks the whole deployment pipeline, and you can't scale just the "hot" part of the app independently. Microservices let teams deploy independently, scale independently, and pick the right tech per service.

My real example

In the sanctions screening API I built, /screen, /forecast, and /explain were designed as logically separate concerns behind one FastAPI app — a natural next step would be splitting /forecast (heavier, ML-style workload) into its own service so it can scale independently of the lightweight /screen lookups, without touching the matching engine's deploy cycle.

Trade-off
Use whenTeams need to deploy independently; different parts of the system have very different scaling needs; the domain has clear bounded contexts.
Avoid whenSmall team, small domain, early-stage product — you'll pay a distributed-systems tax (network calls, data consistency, deployment complexity) before you need the benefit.
Say this in interview

"Microservices solve an organizational scaling problem as much as a technical one — they let teams ship independently. I've worked in a modular monolith with clear service boundaries; the honest trade-off is you're trading deployment simplicity for team autonomy, and that trade only pays off once you actually have multiple teams stepping on each other."

▸ Deep dive: how services actually talk, and what breaks

Communication: Synchronous (REST/gRPC) for request-response needs; asynchronous (message queue/event bus) when the caller doesn't need to wait. Mixing both is normal — sync for reads, async for side-effects like notifications.

What breaks first: Distributed data consistency (no more single DB transaction across services), debugging (a request now spans 5 logs, so you need distributed tracing — correlation IDs, tools like OpenTelemetry/Jaeger), and network reliability (services must handle timeouts, retries, circuit breakers — e.g. Polly in .NET).

Service boundaries: The #1 mistake is drawing boundaries around technical layers (a "database service", a "validation service") instead of business capabilities (Order service, Inventory service). Wrong boundaries cause chatty inter-service calls — a smell interviewers listen for.

Q: How do you handle a transaction that spans two services?
You generally don't use a distributed transaction (2PC is heavy and fragile). Instead use the Saga pattern — a sequence of local transactions with compensating actions if a later step fails.
Q: How do services find each other?
Service discovery (e.g. Consul, Kubernetes DNS/service objects) or an API gateway that routes based on path/host, so callers don't hardcode IPs.

CQRS

Command Query Responsibility Segregation — split how you write from how you read

Define: Separate the model (and often the data path) used to update data (Commands) from the model used to read data (Queries). Writes go through a model optimized for business rules and consistency; reads go through a model optimized for the shape the UI actually needs.

Analogy: A restaurant kitchen (writes) is optimized for correctly preparing food — order, checks, steps, quality control. The menu board (reads) is optimized for a customer glancing at it and deciding fast. You wouldn't force the customer to read the kitchen's ticket system, and you wouldn't run the kitchen off the menu board. Same underlying business (food), two very different shapes of "view" into it.
Why it exists

In many real systems, reads vastly outnumber writes, and the ideal shape for a write (normalized, validated, consistent) is completely different from the ideal shape for a read (denormalized, fast, sometimes eventually-consistent). One model trying to serve both ends up compromised for both.

My real example

In the sanctions API, the "write side" is how a new watchlist gets ingested and normalized across OFAC SDN, OFAC Non-SDN, and UN sources into a consistent internal shape. The "read side" is /screen — a fast lookup path that doesn't care about ingestion rules at all, only about matching speed. I didn't formally implement CQRS with separate stores, but the conceptual split — ingestion/normalization logic vs. lookup/matching logic — is exactly the same idea, and it's the natural next step if read volume ever needs its own scaled, denormalized index.

Trade-off
Use whenRead and write workloads have very different volume or shape needs; complex domain logic on writes but simple, high-volume reads (dashboards, search).
Avoid whenSimple CRUD app — CQRS adds real complexity (often two models, sometimes two data stores, possible eventual consistency) that isn't justified without a real read/write asymmetry.
Say this in interview

"CQRS isn't 'always use two databases' — at its simplest it's just two different models: one shaped for enforcing business rules on write, one shaped for fast, denormalized reads. I've applied the concept even without full event sourcing — separating my ingestion/validation logic from my lookup logic — because coupling them was making both harder to change."

▸ Deep dive: CQRS + Event Sourcing, and eventual consistency

CQRS is often paired with Event Sourcing, where instead of storing current state, you store the sequence of events that led to it (like a bank ledger, not just a balance). The read model is then a "projection" rebuilt from events. This isn't required for CQRS, but they combine well because events naturally feed the read-side projections.

Eventual consistency risk: If read and write use separate stores, there's a lag — a user might write data and not immediately see it on the read side. You must decide if that's acceptable for the use case (analytics dashboard: yes; account balance: usually no, or you need read-your-writes tricks).

Q: Does CQRS require two databases?
No. You can implement CQRS with one database and just two different model/query paths. Two databases is an optimization you add only when the read/write scaling needs actually diverge.
Q: What's the biggest real-world risk with CQRS?
Over-engineering — teams adopt it for apps with no meaningful read/write asymmetry and pay the complexity cost for nothing.

Event-Driven Architecture

Services react to things that happened, instead of being told what to do

Define: Instead of Service A directly calling Service B, Service A publishes an event ("OrderPlaced") to a broker (Kafka, RabbitMQ, SNS/SQS, Azure Service Bus). Any service that cares subscribes and reacts. The publisher doesn't know or care who's listening.

Analogy: A notice board in an office versus phoning every single person individually. When something happens (a policy change), you pin one notice. HR reads it, Finance reads it, Facilities reads it — each acts in their own way, on their own time. You didn't have to know in advance who'd care, and adding a new department later doesn't mean changing your announcement process.
Why it exists

Direct service-to-service calls create tight coupling — Service A must know about B, C, and D, and if any is slow or down, A is affected. Events decouple producers from consumers: producers don't know who's listening, and new consumers can be added without touching the producer at all.

My real example

In the architect/reviewer role on the sanctions API, when a match is found by /screen, the natural extension is publishing a "SanctionsMatchFound" event rather than the screening service directly calling a notification service, a case-management service, and an audit-log service synchronously. Each of those becomes an independent subscriber — I can add an audit consumer later without redeploying the screening engine at all.

Trade-off
Use whenMultiple downstream systems need to react to the same fact; you want producers and consumers to evolve independently; workflows are naturally asynchronous.
Avoid whenYou need an immediate response/result back (checkout confirmation to a user) — pure fire-and-forget events make that awkward; also adds operational overhead (a broker to run, monitor, and reason about).
Say this in interview

"Event-driven design trades immediate consistency and simple call-stacks for loose coupling and independent scalability. The failure mode I watch for is 'event soup' — so many events flying around that no one can trace a business process end-to-end without distributed tracing and good event schemas."

▸ Deep dive: choreography vs orchestration, at-least-once delivery

Choreography: each service reacts to events independently, no central coordinator — highly decoupled but hard to see the "big picture" of a business process. Orchestration: a central coordinator (e.g. a Saga orchestrator) tells each service what to do next — easier to reason about and monitor, but reintroduces a central point of coupling.

Delivery guarantees: Most brokers give "at-least-once" delivery, meaning a consumer might see the same event twice. Your handlers must be idempotent — processing the same "OrderPlaced" event twice should not double-charge or double-ship.

Q: How do you prevent duplicate event processing?
Idempotency keys — track a unique event/message ID and skip processing if you've already handled it, typically via a dedupe table or the message ID unique constraint.
Q: SNS/SQS vs Kafka — when which?
SQS/SNS: simple pub/sub and queueing, fully managed, great for decoupled microservices at moderate throughput. Kafka: when you need replayable event logs, very high throughput, or multiple consumer groups reading the same stream independently at their own pace.

Saga Pattern

A distributed transaction, done as a chain of local transactions with an "undo" plan

Define: A sequence of local transactions across multiple services, where each step publishes an event/triggers the next step. If a step fails partway through, previously completed steps are undone using compensating transactions (not a database rollback — an explicit "reverse" action).

Analogy: Booking a trip: reserve a flight, then a hotel, then a rental car. There's no single "transaction" spanning three different companies' systems. If the car rental fails, you don't get a magic rollback — you explicitly cancel the flight and the hotel. That explicit cancel-if-later-step-fails is exactly what a Saga's compensating transaction is.
Why it exists

In a monolith with one database, a failed step just rolls back the whole ACID transaction. Across microservices with separate databases, there's no shared transaction to roll back. Saga is how you get "all-or-nothing" behavior back, without a fragile distributed transaction (2PC) that locks resources across services.

My real example

I'd frame this from the domain even without having built a full Saga: if screening (/screen), case creation, and notification were three separate services, and case creation failed after a match was already logged, a Saga-style design would fire a compensating "MatchLogReverted" step rather than leaving an orphaned record — I'd flag this explicitly as an architectural improvement I'd propose, not a "did it exactly" claim.

Trade-off
Use whenA business process spans multiple services/databases and needs all-or-nothing semantics without a distributed transaction.
Avoid whenEverything lives in one database (just use a normal ACID transaction) — Saga adds real design complexity (you must design a compensating action for every step) that's wasted if you don't actually have the distributed problem.
Say this in interview

"Saga gives you eventual atomicity, not immediate atomicity — there's a window where the system is in an intermediate state. The hard part isn't the happy path, it's designing a correct compensating action for every single step, including ones that can't be perfectly undone, like 'email already sent.'"

▸ Deep dive: choreography-based vs orchestration-based Saga

Choreography Saga: each service listens for the previous service's event and emits its own — fully decoupled, but hard to track overall progress and easy to create circular dependencies.

Orchestration Saga: a central Saga orchestrator explicitly calls each service and decides what happens on failure — easier to monitor and reason about as one workflow, at the cost of that orchestrator becoming a critical, coupled component.

Most real production systems favor orchestration once the number of steps exceeds 3-4, purely for observability and debuggability.

Q: What if a compensating action itself fails?
You need retries with backoff, and ultimately a manual intervention/dead-letter queue path — some things genuinely can't be perfectly undone (a sent email, a physical shipment), so the design must acknowledge that upfront rather than pretend perfect rollback is possible.

Domain-Driven Design

Model the software around the real business, using the business's own language

Define: An approach where you structure code around the business domain rather than around technical layers, using a Ubiquitous Language (the exact terms the business uses) and dividing large domains into Bounded Contexts — explicit boundaries where a term means one specific thing.

Analogy: The word "Customer" means something different to Sales (a lead with a budget) and to Support (someone with an active ticket history). DDD says: don't force one giant "Customer" class to serve both meanings. Give Sales its own bounded context with its own "Customer" model, and Support its own — and be explicit about how the two map to each other at the boundary, the same way two countries agree on a currency exchange rate rather than sharing one currency.
Why it exists

Large systems built purely around technical layers (controllers, services, repositories) tend to lose the business meaning — the code stops mapping to how the business actually thinks, and every change requires a translation exercise between "what the business asked for" and "which layer does that touch." DDD keeps code and business language in sync, and gives natural seams for microservice boundaries.

My real example

In the sanctions API, "Match" isn't just a data row — it has real domain meaning: a weighted composite of fuzzy, phonetic, and Levenshtein scores, tied to a specific watchlist source (OFAC SDN, OFAC Non-SDN, UN) with its own matching rules. Treating "match against OFAC SDN" and "match against UN list" as the same generic concept would have hidden real business differences — so the config-driven, per-list architecture was effectively a DDD instinct: keep the domain rules explicit and separated by source, not flattened into one generic matcher.

Trade-off
Use whenComplex business domain with real rules and nuance (finance, compliance, insurance, logistics) where getting the model wrong causes real bugs.
Avoid whenSimple CRUD/data-entry apps with no real business logic — the ceremony (aggregates, bounded contexts, ubiquitous language workshops) is overhead you don't need.
Say this in interview

"DDD's real value for me isn't the tactical patterns like Aggregates and Value Objects — it's the discipline of using the business's actual language in the code, and being deliberate about where one term's meaning ends and another begins. That's what naturally gives you good microservice boundaries later."

▸ Deep dive: Aggregates, Value Objects, Bounded Context mapping

Aggregate: a cluster of objects treated as one consistency unit, with one "root" entity controlling access (e.g. an Order aggregate containing OrderLines — you never modify an OrderLine directly, always through the Order).

Value Object: an object defined entirely by its data, with no identity — a Money(amount, currency) or an Address is equal to another with the same values, unlike an Entity which has identity even if its data changes.

Context Mapping: when two bounded contexts need to talk, you define the relationship explicitly — e.g. an Anti-Corruption Layer that translates one context's model into another's, so a change on one side doesn't silently corrupt the other's meaning of a term.

Q: How does DDD relate to microservices?
Bounded Contexts are a strong candidate for microservice boundaries — each bounded context can become its own service, since it already has its own model and language, minimizing cross-service chatter.
