OAuth2 vs OIDC

Authorization vs authentication — a distinction almost everyone blurs, including in interviews

Define: OAuth2 is an authorization framework — it lets an app get limited access to a resource on a user's behalf, without ever seeing their password ("let this app read my Google Calendar"). OIDC (OpenID Connect) is a thin identity layer built on top of OAuth2, adding actual authentication — proving who the user is, via a standardized ID Token.

Analogy: OAuth2 alone is a hotel key card scoped to specific rooms — it proves you're allowed into the gym and the pool, but it doesn't actually identify who you are, just that you were granted access. OIDC adds a photo ID check on top — now the hotel also knows and can verify exactly who's holding that card.
Why the distinction matters

A very common mistake: using a plain OAuth2 access token to identify a user ("this token exists, so someone's logged in") when OAuth2 was never designed to guarantee that — access tokens are opaque authorization grants, not identity proofs. OIDC's ID Token is the piece specifically designed and signed for identity verification.

Trade-off
Use OAuth2 alone whenYou genuinely only need delegated access to a resource, with no need to know or verify who the user actually is.
Use OIDC whenYou need actual "who is this user" login — which is nearly every consumer-facing "Sign in with Google/Microsoft" flow, even though people often just say "OAuth login" loosely.
Say this in interview

"I'm precise about this distinction because it's a real source of bugs — OAuth2 answers 'is this app allowed to do X,' OIDC answers 'who is this person.' If a login flow is built on bare OAuth2 access tokens without OIDC's ID token, you don't actually have verified identity, just a valid-looking access grant."

JWT Internals

Signed, not encrypted — the single most important thing to know about a JWT

Define: A JSON Web Token has three parts — header.payload.signature — base64url-encoded and concatenated. The signature proves the token wasn't tampered with since issuance, verified with a secret (HMAC) or a public/private key pair (RSA/ECDSA).

The critical misconception
JWTs are signed, NOT encrypted by defaultThe payload is just base64-encoded — trivially readable by anyone, including the end user holding it. Never put sensitive data (passwords, SSNs, secrets) directly in a standard JWT payload. If you truly need the payload itself hidden, that requires a separate encryption step (JWE), which is uncommon in practice — the standard pattern is simply "don't put secrets in the payload," not "encrypt the JWT."
Analogy: A JWT is like a sealed wax-stamped letter with the contents printed in clear, large text visible through the envelope — the wax seal (signature) proves nobody tampered with the letter after it was sealed, but it does nothing to hide what's written inside. Anyone holding the letter can read every word.
Trade-off — JWT vs opaque session tokens
JWT (stateless)Self-contained — any service holding the public key can verify it without a database lookup, great for distributed/microservices auth. Downside: hard to revoke before expiry, since there's no central "is this still valid" check by default.
Opaque session tokenJust a random ID, looked up against a server-side store on every request — trivially revocable (delete the server-side record), but requires that lookup on every request, and doesn't scale as cleanly across independent services without a shared session store.
Say this in interview

"The thing I always flag about JWTs is that they're signed, not encrypted — the payload is readable by anyone holding the token, so nothing sensitive belongs in it. The revocation trade-off is the other big one: JWTs are great for stateless, distributed verification, but genuinely hard to revoke before expiry without adding back some form of server-side check, which partially undoes the statelessness benefit."

Encryption — At Rest & In Transit

Two different protections, addressing two different threats
Encryption in transit (TLS)Protects data while it's moving across a network — prevents an attacker on the network path from reading or tampering with it. This is what HTTPS provides.
Encryption at restProtects data while it's stored — on disk, in a database, in a backup. Protects against a stolen disk, an unauthorized database dump, or a misconfigured backup exposure. TLS alone does nothing for this — data can be perfectly protected in transit and still sit as cleartext once it lands on disk.
Symmetric vs Asymmetric — the practical distinction

Symmetric (AES): one shared key encrypts and decrypts — fast, used for the bulk of actual data encryption. Asymmetric (RSA/ECDSA): public key encrypts (or verifies), private key decrypts (or signs) — slower, but solves the "how do two parties who've never met securely share a key" problem. In practice, TLS uses asymmetric crypto briefly during the handshake specifically to safely exchange a symmetric session key, then switches to fast symmetric encryption for the actual data — combining both for the reason each is good at.

Say this in interview

"I treat encryption in transit and at rest as two separate requirements, not one checkbox — TLS covers the network, but if the database itself is compromised or a backup leaks, encryption at rest is what actually protects the data. I'd expect both, by default, on anything handling sensitive data — not one as a substitute for the other."

OWASP Top 10 — the Ones Worth Knowing Cold

Not memorizing all 10 by number — knowing the two or three that come up constantly
Injection (SQL Injection)

Define: Untrusted input concatenated directly into a query lets an attacker alter the query's logic. Fix: Parameterized queries / prepared statements — always, no exceptions — which both EF Core and JPA/Hibernate do by default when you use them correctly, meaning most modern ORM usage is already protected unless someone drops to raw string-concatenated SQL.

Broken Access Control

Define: The system correctly identifies who a user is, but fails to properly check what they're allowed to do — e.g. an API that checks a JWT is valid but never checks whether this specific user is allowed to access this specific resource (an "insecure direct object reference" — changing an ID in a URL to access someone else's data). Consistently ranked as one of the most common real-world vulnerabilities, precisely because authentication is easy to remember and authorization-per-resource is easy to forget.

Cross-Site Scripting (XSS)

Define: Untrusted input rendered into a page as if it were trusted HTML/JS, letting an attacker run script in another user's browser session. Fix: Output encoding/escaping by default — modern frameworks like Angular and React escape by default, which is exactly why bypassing that default (Angular's [innerHTML]/bypassSecurityTrust*, React's dangerouslySetInnerHTML) is a real red flag worth scrutinizing in code review.

Say this in interview

"Rather than reciting all 10 by number, I'd flag Broken Access Control as the one I actually see most in practice — teams get authentication right (is this a valid user) and then forget the second check (is this valid user allowed to touch this specific resource). That gap is usually where real vulnerabilities live, more than exotic injection attacks against a modern ORM."

Zero Trust

Stop assuming anything inside your network perimeter is automatically safe

Define: A security model that assumes no implicit trust based on network location — every request is authenticated and authorized, whether it originates from outside the network or from another service sitting right next to it inside the same data center. Contrasts with the older "castle and moat" model, where anything inside the firewall was implicitly trusted.

Analogy: A castle-and-moat model trusts anyone who's already inside the castle walls, no further questions asked. Zero trust is more like a modern office building where your badge is checked at every single door — the lobby, the elevator, your floor, even a meeting room — not just once at the front entrance, because someone getting past the front door doesn't mean they should have access to everything behind it.
Why it exists

The perimeter model breaks down once you have microservices, cloud infrastructure, and remote work — there often isn't one clear "inside the firewall" anymore, and a single compromised internal service or credential in a perimeter model can move laterally with little friction. Zero trust assumes breach is possible anywhere and checks identity/authorization at every hop.

My real example — where this connects to what I already do

Requiring every microservice-to-microservice call to carry and validate its own auth token — rather than trusting "it's an internal call, it must be fine" — is zero trust applied practically, and it's the same instinct behind APIM/API Gateway enforcing auth centrally rather than assuming internal traffic is automatically safe.

Say this in interview

"Zero trust is really just extending the same authorization discipline you'd apply to external traffic to internal service-to-service calls too — not assuming 'it's coming from inside our network, so it's fine.' At a practical level, that means every service call carries and validates its own credentials, not just the request that first entered the system."

Rapid Recall Drill

OAuth2 vs OIDC, in one line?
OAuth2 = authorization (can this app do X); OIDC = authentication (who is this user), built on top of OAuth2.
Is a JWT payload encrypted?
No — signed, not encrypted. Base64-encoded and readable by anyone holding it; never put secrets in the payload.
Why can't you easily revoke a JWT before expiry?
It's self-contained/stateless by design — no central "is this still valid" check unless you deliberately add one back, which partially undoes the statelessness benefit.
Does TLS protect data sitting in a database?
No — TLS covers data in transit only; encryption at rest is a separate requirement for data on disk/in backups.
Most common real-world OWASP issue in practice?
Broken Access Control — auth (who are you) is usually implemented correctly; per-resource authorization (are you allowed to touch THIS record) is what gets forgotten.
Zero trust in one line?
No implicit trust based on network location — every request, internal or external, is authenticated and authorized.
