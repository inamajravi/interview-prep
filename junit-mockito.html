JUnit 5 Basics

Same xUnit patterns, different annotations
[Fact] / [Theory] / [InlineData] ↔ @Test / @ParameterizedTest / @ValueSource
xUnit (.NET) 
[Theory]
[InlineData("SDN", true)]
[InlineData("UN", true)]
public void Screen_KnownList_Matches(
    string list, bool expected) {
  var result = _svc.Screen(list, "John Doe");
  Assert.Equal(expected, result.IsMatch);
}
JUnit 5 
@ParameterizedTest
@ValueSource(strings = {"SDN", "UN"})
void screen_KnownList_Matches(String list) {
  var result = svc.screen(list, "John Doe");
  assertTrue(result.isMatch());
}

Lifecycle mapping: [assembly: ...] setup/teardown → @BeforeEach/@AfterEach (per test) and @BeforeAll/@AfterAll (once per class, must be static unless @TestInstance(PER_CLASS) is set — a small gotcha .NET devs hit since xUnit doesn't require statics for the equivalent).

Say this in interview

"JUnit 5's annotation model maps almost directly onto xUnit — Fact/Theory become Test/ParameterizedTest. The one habit I'd build fast is JUnit 5's per-class vs per-method test instance lifecycle, since it changes whether BeforeAll needs to be static."

Mockito

Same mocking philosophy as Moq — arrange, act, assert, verify
Mock<T> + Setup + Verify ↔ @Mock + when(...).thenReturn(...) + verify(...)
Moq (.NET) 
var mock = new Mock<IMatcher>();
mock.Setup(m => m.Score(It.IsAny<string>()))
    .Returns(0.95);
var svc = new ScreeningService(mock.Object);
svc.Screen("John Doe");
mock.Verify(m => m.Score("John Doe"), Times.Once);
Mockito 
IMatcher mock = mock(IMatcher.class);
when(mock.score(anyString())).thenReturn(0.95);
ScreeningService svc = new ScreeningService(mock);
svc.screen("John Doe");
verify(mock, times(1)).score("John Doe");
Analogy: Both frameworks are hiring a stunt double for a scene — you tell the double exactly what to do when cued (Setup/when-thenReturn), then check afterward the director actually called on them the right number of times (Verify).
Trade-off / common trap
Over-mockingMocking too deeply (mocking value objects, or mocking classes you own instead of testing them directly) is the same anti-pattern in both ecosystems — mock at architectural seams (external dependencies, interfaces), not everywhere.
Say this in interview

"Mockito's when/thenReturn plus verify is functionally identical to Moq's Setup/Verify — same arrange-act-assert discipline. The judgment call that matters more than syntax is where to mock: at real architectural seams, not on every collaborator, or your tests end up just testing your mocks."

Spring Boot Test Layers

A genuinely new idea worth understanding well — sliced tests that load only part of the app

Define: Spring Boot Test provides annotations that load a partial application context tailored to a layer, instead of the whole app, making tests both faster and more focused:

  • @WebMvcTest — loads only the web layer (controllers, filters), mocks everything below it. Roughly equivalent to testing an ASP.NET Core controller with `TestServer`/`WebApplicationFactory` but scoped even narrower.
  • @DataJpaTest — loads only JPA repositories against an in-memory or test database, rolling back after each test.
  • @SpringBootTest — loads the full application context, closest equivalent to a true .NET integration test using `WebApplicationFactory`.
Analogy: Testing a car's brakes doesn't require driving the whole car on a highway — you put just the brake assembly on a test rig. Slice tests are exactly that: load only the "assembly" you're testing, not the whole running app.
Why it exists

Loading the full Spring context for every test is slow — hundreds of controller tests each booting the whole app adds up fast. Slice annotations keep the fast majority of tests fast, reserving full-context @SpringBootTest for genuine end-to-end integration tests.

Trade-off
Use slices whenTesting one layer's logic in isolation — fast feedback, most of your test suite should live here.
Use full @SpringBootTest whenYou genuinely need to verify layers wired together correctly — sparingly, since it's the slowest test type.
Say this in interview

"This is the one piece of Spring testing that's genuinely different from .NET's WebApplicationFactory-centric integration testing — slice annotations like @WebMvcTest and @DataJpaTest let most of the test suite stay fast by loading only the relevant layer, reserving full @SpringBootTest for real integration coverage."

Testcontainers

Real Postgres/Kafka/Redis in a Docker container, spun up just for the test run

Define: A library that spins up real dependencies (a real Postgres, real Kafka, real Redis) inside lightweight Docker containers, scoped to your test run, then tears them down — instead of mocking the database or relying on a flaky shared test environment.

Analogy: Instead of rehearsing a play with a stand-in prop sword, you rehearse with the actual sword for one rehearsal — safely, in a controlled setting, then it goes back in the case. You get real behavior, not an approximation, without permanent risk.
Why it exists

In-memory fakes (like H2 standing in for real Postgres) can behave subtly differently from production databases — different SQL dialect quirks, different constraint behavior. Testcontainers closes that gap by testing against the real thing, just ephemeral and isolated.

Trade-off
Use forIntegration tests where DB-specific behavior matters (constraints, migrations, real SQL dialect quirks).
CostRequires Docker in the CI environment and is slower than in-memory fakes — reserve for integration-level tests, not every unit test.
Say this in interview

"Testcontainers solves the 'my tests pass against H2 but fail against real Postgres in production' problem — it trades a bit of speed and a Docker dependency in CI for testing against the real database engine. I'd use it for integration-level DB tests, not as a blanket replacement for fast unit tests."
